### Workstream 1 — Recent files (welcome screen)

The welcome screen currently shows only keyboard shortcuts. Make it instantly useful by showing a clickable grid of the last ~10 opened files.

**1a — Recents store (`src/lib/persistence.js`)**
Add two functions mirroring `saveSession`/`loadSession`, backed by a new `mdpeek-recents` localStorage key. Shape: `[{ path, name, openedAt }]` (ISO timestamp). `addRecent(path)` helper trims to 10, dedupes by path, moves the most-recent to the top.

**1b — Record on open (`src/main.js`)**
In `openPath()` (line 428), after `store.open(...)`, call `addRecent(path)`. Skip for `path === null` (untitled tabs) and PDFs/Excalidraw are fine to record (they're real files the user opened).

**1c — Render recents in the welcome screen (`src/main.js`)**
Make `WELCOME_HTML` a function `renderWelcome()` that:
- Returns the static welcome markup (logo, title, hints) — unchanged.
- If `loadRecents()` returns any entries, appends a `<section class="recent-files">` after the hints: a heading "Recent" + a list of clickable rows (file icon by type — reuse the MD/PDF/EX/`{}` logic from `tabs.js` iconFor, or simpler: just a generic file glyph + name + path-ellipsis). Each row has `data-path` so a delegated click handler opens it.
- Click handler (delegated on `el.document` for `.recent-item` clicks): reads `data-path` and calls `openPath` after reading the file via `invoke('read_file', { path })`. Wrap in try/catch — if the file was deleted/moved since it was opened, show a toast and remove it from recents.

**1d — Update all 4 welcome render call sites** (lines 252-253, 411-412, 1483-1485, 1497-1499) to call `renderWelcome()` instead of referencing the `WELCOME_HTML` constant. Convert the constant to a function.

**1e — CSS (`src/styles/base.css`, near the existing `.welcome` rules ~line 590)**
- `.recent-files` — margin-top: 32px, max-width: 520px, centered.
- `.recent-title` — same uppercase-tracked style as `.setting-section-title` (11px, fg-muted).
- `.recent-item` — flex row, padding 8-10px, border-radius, hover background; `.recent-name` (13px, fg, truncate with ellipsis), `.recent-path` (11px, fg-muted, ellipsis).

### Workstream 2 — Word count + reading time (status line)

A slim status bar at the bottom of the editor pane showing word count, character count, and estimated reading time. Markdown writers check this constantly.

**2a — Status bar markup (`index.html`)**
Add a `<div id="editor-status" class="editor-status" aria-live="polite"></div>` as the last child inside `.editor-pane` (after `.editor-wrap`, line 187). Styled `flex: 0 0 auto` so it sits at the bottom of the editor half.

**2b — Count logic (`src/main.js`)**
Add a `wordCount(text)` helper that:
- Strips markdown syntax (fences, headings `#`, emphasis `*_/`, links `[text](url)`, images, HTML).
- Counts CJK characters as individual words (each ideograph = 1 word) plus Latin words split on whitespace. This is the standard for mixed CJK/Latin counting.
- Returns `{ words, chars, readMins }` where `readMins = Math.max(1, Math.round(words / 200))`.

**2c — Update on input (`src/main.js`)**
In the existing editor `input` listener (line 1165), after `markDirty`, call `updateEditorStatus()`. Also call it once after `doc.editor.setValue(doc.content)` in the edit-mode render branch (line 358) so the status is correct when entering edit mode. Hide the status bar (`hidden` class) when not in edit mode — add the toggle in `renderActive` where edit mode is shown/hidden.

**2d — Render the status (`src/main.js`)**
`updateEditorStatus()` reads the active doc's editor value (via `doc.editor.getValue()` if `doc.editor` exists, else `el.editor.value`), runs `wordCount`, and sets `el.editorStatus.innerHTML` to `<span>423 words</span><span>2,184 chars</span><span>~2 min read</span>`. Use tabular-nums + bullet separators. No-op if `el.editorStatus` is hidden (not in edit mode).

**2e — CSS (`src/styles/base.css`, near `.pane-header` ~line 338)**
- `.editor-status` — `flex: 0 0 26px; display: flex; gap: 14px; padding: 0 16px; align-items: center; font-size: 11.5px; color: var(--fg-muted); border-top: 1px solid var(--border-subtle); font-variant-numeric: tabular-nums; user-select: none;`.
- `.editor-status.hidden { display: none; }`.

### Workstream 3 — Focus / Zen mode

Hide the header + sidebar for distraction-free reading. One keystroke in, one out.

**3a — Toggle (`src/main.js`)**
Add `toggleFocus()` mirroring `toggleSidebar`:
```js
function toggleFocus() {
  const on = document.body.classList.toggle('focus-mode');
  localStorage.setItem('mdpeek-focus', on ? '1' : '0');
}
```
Wired to two key bindings:
- **F11** — new capture-phase listener (mirror the F3 pattern at line 1217): `if (e.key === 'F11') { e.preventDefault(); toggleFocus(); }`. preventDefault because Tauri WebView2 may otherwise use F11 for native fullscreen.
- **Escape** — when focus mode is on, Escape turns it off. Add to the existing global Escape handling (or a new check at the top of the keydown handler): `if (e.key === 'Escape' && document.body.classList.contains('focus-mode')) { toggleFocus(); return; }`.

**3b — Restore on init (`src/main.js`)**
Mirror the sidebar restore at line 1419: `if (localStorage.getItem('mdpeek-focus') === '1') document.body.classList.add('focus-mode');`

**3c — CSS (`src/styles/base.css`)**
```css
body.focus-mode .app-header { display: none; }
body.focus-mode #toc { display: none; }
```
Because `body` is `display: flex; flex-direction: column` and main is `flex: 1`, hiding the header (which is `flex: 0 0 52px`) makes the document area fill the full height automatically — no layout math. A subtle slide-in transition on the header is optional polish.

**3d — Welcome-screen hint update (`index.html` or renderWelcome)**
Add a hint chip: `<kbd>F11</kbd> Focus mode` so it's discoverable.

### Workstream 4 — Export to HTML (self-contained)

One button + command that bundles the rendered markdown + the app's CSS + the active hljs theme into a single `.html` file. Works fully offline.

**4a — Rust command (`src-tauri/src/commands.rs`)**
Add `save_file_as_html(content: String) -> Result<String, String>` mirroring `save_file_as` (line 50), but with `.add_filter("HTML", &["html", "htm"])`, `.set_file_name("exported.html")`. Returns the saved path or rejects with `"cancelled"`. Register it in `lib.rs` invoke_handler (line 131).

**4b — Export logic (`src/main.js`)**
Add `exportHtml()`:
1. Bail if no active doc or the doc isn't markdown/plain (PDF/Excalidraw/code don't export to HTML — they have their own formats).
2. Render the markdown via `renderMarkdown(doc.content)` (already imported-adjacent; need to add `renderMarkdown` to the import from `renderer.js` — it's currently only used inside `viewer.js`).
3. Build the CSS to inline:
   - Read the active theme tokens from the matching `:root[data-theme="..."]` block — simplest: grab `getComputedStyle(document.documentElement)` for the key vars (`--bg`, `--fg`, `--content-font-family`, `--content-line-height`, etc.) and emit a `:root { ... }` block.
   - Inline the **content.css** rules (markdown body styling) — fetch the stylesheet text. Since content.css is bundled by Vite (not a stable URL at runtime), the cleanest approach is to **read CSS rules from the document's stylesheets** via `document.styleSheets` (find the one whose rules match `.markdown-body`), or simpler: **ship a constant string** `MARKDOWN_CSS` containing the prose styles (headings, code, lists, alerts, tables) hand-curated for export. The constant-string approach avoids runtime stylesheet crawling and works offline. I'll go with the constant.
   - Inline the **active hljs theme**: read `HLJS_FOR_THEME[dataset.theme]`, get that `<link>`'s `.href`, `fetch(href).then(r => r.text())`. The links are CDN URLs — fetch will succeed online. If fetch fails (offline), fall back to a minimal default (leave code uncolored — still readable). Wrap in try/catch.
4. Assemble the HTML document:
   ```
   <!DOCTYPE html><html><head><meta charset><title>filename</title>
   <style> /* :root vars + markdown css + hljs css */ </style></head>
   <body class="markdown-body"> /* rendered HTML */ </body></html>
   ```
5. Call `await invoke('save_file_as_html', { content })`. On the `'cancelled'` rejection, silently return. On success, toast "Exported to HTML".
6. Title the file from the doc's basename (reuse the existing `basename()` helper at line 111).

**4c — Toolbar button (`index.html`)**
Add an export button inside the first `.tool-group` (after `#btn-sidebar`, line 78): `<button id="btn-export" class="tool-btn icon-only" title="Export as HTML" aria-label="Export as HTML">` with a download/share SVG icon. Hidden by default (add `hidden` class) — only shown for markdown docs (toggle in `renderActive`, near where the draw button is shown/hidden for PDFs).

**4d — Wire the button (`src/main.js`)**
`el.export.addEventListener('click', exportHtml)` in the events block. Add `export: document.getElementById('btn-export')` to the `el` object. Show/hide in `renderActive`: visible for markdown docs in view mode, hidden for PDF/Excalidraw/code/plain/welcome.

**4e — Shortcut (optional, low priority)**
Ctrl+Shift+E is a natural fit but it conflicts with nothing currently (Ctrl+E is plain toggle, Shift is a modifier the existing handler ignores). Add `if (e.shiftKey && k === 'e') { e.preventDefault(); exportHtml(); }` to the keydown handler. Optional — the toolbar button is the primary affordance.

### Out of scope
- Recent files from across reboots only (not "currently open tabs" — that's the session).
- Reading-time for code/PDF/Excalidraw (markdown + plain text only).
- Exporting PDF or Excalidraw to HTML (they're not markdown).
- Animated transitions for focus mode (instant hide/show is cleaner and simpler).
- Configurable reading-speed (200 wpm default is the widely-accepted standard).
- Export theme picker (uses the active theme — keeps it one-click).
