## Plan: PDF text selection, in-PDF search, and drawing annotations

Three features, all building on the PDF viewer added in v0.6.0. They share a common foundation (per-page wrapper structure + text layer), so they're implemented together.

### Foundation: text layer + restructured page wrapper

The current page structure is `.pdf-page > canvas`. I'll restructure each page to a three-layer stack:

```
.pdf-page (position: relative; sized to viewport)
  ├── canvas          (z-index: 1 — page render, existing)
  ├── .textLayer div  (z-index: 2 — transparent text for selection + search)
  └── canvas.pdf-draw (z-index: 3 — annotation drawing, pointer-events toggled by mode)
```

**Text layer rendering** — in `renderPage()`, after the canvas render completes, call `page.getTextContent()` and construct a `TextLayer({ textContentSource, container, viewport })`, then `await textLayer.render()`. Store each `TextLayer` instance in a `Map<pageNum, TextLayer>` on the controller so search can access `textDivs`.

**Essential `.textLayer` CSS** — instead of importing pdf.js's 163 KB `pdf_viewer.css`, I'll port only the ~40 lines that matter (transparent spans, absolute positioning, `::selection` styling, `color-scheme: only light`) into `content.css`. The spans are invisible (`color: transparent`) but selectable — the browser's native selection "just works" over the canvas.

### Feature 1: Text selection (#2 in user's list)

**Zero additional code** once the text layer exists. The `.textLayer` spans have `user-select: text` and `cursor: text`, so dragging selects PDF text natively. Copy works via `Ctrl+C`. The `::selection` style shows a blue highlight over the canvas.

The one fix needed: currently `.pdf-page` has no `position: relative`, so the absolute-positioned text layer won't anchor correctly. Add `.pdf-page { position: relative; }` to `content.css`.

### Feature 2: In-PDF search (#1 in user's list)

**`find-bar.js` gets a `'pdf'` mode branch.**

1. `main.js` — change `getMode()` to return `'pdf'` when the active doc is a PDF (`d.pdf ? 'pdf' : d.mode`). Add a `getPdf: () => _activePdf` accessor.
2. `pdf-viewer.js` — extend the `showPdf` controller to expose `pdfDoc` and the `textLayers` Map, so the find bar can search.
3. `find-bar.js` — add `runPdf()` and `stepPdf()`:
   - **`runPdf()`** — loop pages 1..N, call `pdfDoc.getPage(n).getTextContent()`, filter items to those with `.str`, concatenate into one string with a page-boundary map (offset → {pageNum, localOffset}). Run `findMatches()` (reused from `editor-logic.js`) on the flat string. Highlight matches by overlaying a `.pdf-search-highlight` div on the text layer's spans (position calculated from the match's text span positions). Show count `n/total`.
   - **`stepPdf()`** — advance `matchIdx`, re-style the current match's highlight (accent background vs soft), scroll the match into view via the text span's `getBoundingClientRect()`.
   - Match highlighting uses absolutely-positioned `<div>`s over the `.textLayer` (the text spans have known positions; I compute the union rect of the matched character range). The current match gets `--accent` background; others get `--accent-soft`.
4. Search is **async** (getTextContent per page is a round-trip). On first search of a PDF, show a "Searching…" state in the count; cache the per-page text content in the controller so subsequent searches (next/prev, query refinement) are instant.

### Feature 3: Drawing annotations (#3 in user's list) — session-only

**A floating toolbar + per-page annotation canvases.**

**Toolbar** — a `.pdf-draw-toolbar` overlay (modeled on `#find-overlay`), positioned top-center of the document pane, shown only when a PDF tab is active. Contains:
- **Pen** tool (freehand, solid stroke, ~2px, color from a 5-color palette)
- **Highlighter** tool (semi-transparent, thick ~12px, round caps, multiply blend)
- **Eraser** tool (removes strokes that intersect the click point)
- **Color palette** — 5 swatches (black, red, blue, green, yellow) shown when a tool is active
- **Clear all** button
- **Close** button (exits draw mode, returns to normal pointer/select mode)

The toolbar is **modal**: when draw mode is active, the `.pdf-draw` canvases get `pointer-events: all` and the `.textLayer` gets `pointer-events: none` (no text selection while drawing). When draw mode is off, annotation canvases are `pointer-events: none` and text selection works normally.

**Drawing implementation:**
- Each `.pdf-page` gets a `canvas.pdf-draw` sized to the page's CSS dimensions (matching the text layer).
- Strokes are stored as arrays of `{x, y}` points in **page-relative CSS coordinates** (so they survive zoom — re-rendered at the new scale on zoom).
- The controller holds `strokesByPage: Map<number, Stroke[]>`. A `Stroke` is `{ tool: 'pen'|'highlighter', color, width, points: [{x,y}] }`.
- Pointer events on the draw canvas: `pointerdown` starts a stroke, `pointermove` appends points + draws incrementally to the canvas, `pointerup` finalizes. Eraser does hit-testing against existing strokes (point-in-bounding-box) and removes matches.
- **Session-only:** strokes live in the controller's closure. On tab switch / close, `destroy()` discards them (per the user's choice — no persistence). The `Clear all` button empties all pages.

**Zoom interaction:** on zoom change, the page canvases re-render at the new scale (existing behavior), and the annotation layer re-renders all strokes scaled to the new viewport. The stroke coordinate space is normalized (0..1 relative to page dimensions) so scaling is a simple multiply.

### Interaction details
- **Default mode on a PDF tab:** normal pointer — text selection works, scroll works, Ctrl+F searches the PDF.
- **Draw mode:** click the pen/highlighter/eraser button → annotation canvases activate, text selection is disabled. Click "Close" or the active tool again → back to normal.
- **Find on a PDF:** Ctrl+F opens the same find bar, but it dispatches to `runPdf()`. Results show as highlighted overlays. Next/prev scrolls through matches across pages.
- **Eraser:** click on a stroke to remove it (hit-tests the stroke's bounding box). Not a pixel-eraser.

### Edge cases
- **Text layer not yet rendered when search runs** — `runPdf()` awaits the text-layer render for each page before searching it; pages not yet rendered (below the fold) are searched via `getTextContent()` directly (doesn't need the DOM layer).
- **Search across page boundaries** — a match can't span pages (each page's text is independent). The flat-string approach with page-boundary map handles this naturally.
- **Zoom after drawing** — strokes re-render at the new scale. Stroke coordinates are normalized (0..1), so re-rendering is a scale multiply.
- **Very large PDFs** — search caches per-page text content in the controller after first extraction, so next/prev is instant. The first search shows "Searching…" while pages are extracted.
- **Drawing on a page not yet rendered** — the draw canvas exists for every page (created upfront), so you can draw even on pages below the fold. The canvas is sized to the page's standard dimensions.
