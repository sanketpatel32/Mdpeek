## Plan: Unified, idempotent Ctrl+F find across view + edit mode

**Goal:** One Ctrl+F that works everywhere. Single floating find bar, one set of listeners (idempotent — pressing Ctrl+F repeatedly never stacks handlers or duplicates state). Matches the four extras you picked.

### Core design: a single global find module

The current find bar lives inside `#edit-mode` and is wired/destroyed per tab — that's the root cause of fragility. The new design moves find out of `editor.js` entirely into a **new standalone module** `src/views/find-bar.js` with one DOM element appended to `<body>` (outside both mode containers), created once at startup, with listeners attached exactly once.

```
src/views/find-bar.js   ← new file, the whole feature lives here
  exports: initFindBar({ getMode, getEditor, getDocument, toast })
           → { open(), close(), toggle(), findNext(), findPrev() }
```

`main.js` calls `initFindBar(...)` once at startup, passes in accessor callbacks (so the module can ask "what's the active mode?", "give me the active editor instance?", "give me the #document element"), and receives the public API. Then:
- The global keydown handler in `main.js` adds `f` → `find.toggle()`, `g` → `find.findNext()`, plus a separate non-Ctrl handler for `F3`/`Shift+F3`.
- `editor.js`'s Ctrl+F handler is **removed** (the global one catches it via capture phase). All find logic leaves `editor.js`.

### Idempotency guarantees (the key requirement)
1. The find bar DOM element is created **once** (`document.createElement` in module scope, guarded by a `created` flag). `initFindBar` called twice is a no-op on the second call.
2. Event listeners attached **once** on the bar (keydown, input, click on prev/next/close/case buttons). They live for the app lifetime — never torn down, never re-added.
3. `toggle()`/`open()` never clone or rebuild the bar — they only flip the `.hidden` class and focus the input. Repeated Ctrl+F is just "focus + select-all."
4. State (query, caseSensitive, last matches) lives in module scope, reset only on input or close.
5. F3/Ctrl+G after close reuses `lastQuery` and re-runs against the current doc — no stale listener.

### How find works in each mode

**View mode** (the big addition — currently zero search):
- Use `findMatches` (reuse the pure function from `editor-logic.js`) over the **rendered text content** — extracted by walking `#document` text nodes with a `TreeWalker` and building a string + a parallel offset-mapping so character offsets map back to DOM text nodes.
- Actually simpler & more robust approach: walk text nodes once, for each match range, split the text node and wrap the matched substring in `<mark class="find-match">`. The current match gets `<mark class="find-match find-current">`.
- `<mark>` styling: subtle background tint (`--accent-soft`); current match gets accent border + stronger background.
- Prev/next: scroll the current `<mark>` into view (`scrollIntoView({block:'center'})`) and update the classes.
- On close (or input change, or mode switch): clear all marks via a `TreeWalker` that unwraps them (replace each `<mark>` with its text node child, then `normalize()`).
- **Critical:** highlight.js's nested spans inside `<pre><code>` are descended into by the text walker (matches inside code blocks work). Mermaid SVG text is skipped (text nodes inside SVG are filtered out).
- Re-run on input: clear marks, re-walk, re-wrap, jump to nearest match. Debounced 120ms so typing is smooth on big docs.

**Edit mode** (textarea — can't highlight inline, so keep selection-based):
- Reuse the existing approach but route it through the new module: `findMatches` over `editor.getValue()`, `setSelectionRange` for the current match, scroll-to-line via the existing line-height math. Same `jumpTo` logic, just moved into the new module and calling the editor instance via the accessor.
- Matches are NOT visually highlighted (textarea limitation — known/accepted), but the count shows `n/total` and the current match is selected + scrolled into view.
- The old in-`editor.js` find code (openFind/closeFind/runFind/jumpTo/onFindKey/onFindInput/updateCount, ~90 lines) is **deleted**; the exports `openFind`/`closeFind` are removed from the editor API.

### The four extras
1. **Case-sensitive toggle** — an `Aa` button in the bar. `caseSensitive` state in module scope, persisted to `localStorage['mdpeek-find-case']`. Toggling re-runs the search. Button has an `.active` class when on (accent color). Wired once.
2. **Seed query from selection** — on `open()` in edit mode, read `editor.getState()` selection; if non-empty and on a single line, pre-fill the input with the selected text and run the search immediately. In view mode, read `window.getSelection()` — if it's a non-empty text selection inside `#document`, seed from it. If the bar is already open, Ctrl+F does NOT re-seed (just focuses — standard browser behavior).
3. **F3 / Ctrl+G repeat** — global keydown: F3 (with Shift = backward), Ctrl+G (forward), Ctrl+Shift+G (backward). If the bar is open, the bar's own keydown handles F3. If closed, the global handler calls `find.findNext(lastQuery)` / `find.findPrev(lastQuery)` — re-runs against current doc using the stored last query. No-op if no last query.
4. **Esc clears then closes** — bar's keydown: first Esc with non-empty input → clear input + clear highlights + reset count, keep focus; second Esc (empty input) → `close()`. If input already empty on first Esc, close immediately.

### Find bar markup (new global element, appended to body)
```html
<div id="find-overlay" class="find-overlay hidden">
  <div class="find-bar">
    <button class="find-toggle" id="find-case" title="Match case (Aa)" aria-label="Match case" aria-pressed="false">Aa</button>
    <input type="text" class="find-input" placeholder="Find…" spellcheck="false" />
    <span class="find-count">0/0</span>
    <button class="find-prev tool-btn icon-only" title="Previous (Shift+Enter / Shift+F3)">…chevron-up…</button>
    <button class="find-next tool-btn icon-only" title="Next (Enter / F3)">…chevron-down…</button>
    <button class="find-close tool-btn icon-only" title="Close (Esc)">…x…</button>
  </div>
</div>
```
Positioned `fixed` top-right (just under the toolbar), `z-index` above content but below modals. Existing `#find-bar` inside `#edit-mode` in index.html is **removed** (the new one is created by JS).

### Edge cases handled
- **Mode switch while bar open:** `main.js` calls `find.close()` inside `toggleMode()` before re-rendering — prevents textarea selection from a closed editor lingering, and clears view-mode `<mark>`s before the doc is hidden.
- **Tab switch while bar open:** `renderActive()` calls `find.close()` when `_lastRenderedId` changes — no stale highlights on a different doc.
- **External file reload (file-changed event):** view-mode doc is re-rendered via `showDocument`, wiping any `<mark>`s. If the bar is open, `main.js` re-runs `find.run()` after the re-render so highlights refresh on the new content.
- **Empty query:** no marks applied, count shows `0/0`, prev/next disabled.
- **No matches:** count `0/0`, input gets a subtle red outline (`.find-input.no-match`), no marks.
- **Wraparound:** when you hit Next on the last match, it wraps to match 0 (existing `nextMatchIndex` behavior) — a tiny "Wrapped" indicator could be added but I'll keep the count display as feedback to stay simple.
