## Plan: Theme refinement + new themes + reading comfort

Three workstreams the user approved. Plus the answer to "is it still lightweight": **yes — the installer is 4.00 MB** (Tauri/Rust binary is tiny; Typora/Obsidian/VS Code are 80–90 MB). The 528 KB frontend JS chunk is larger than ideal but the user chose to leave it, so no bundle work.

### Workstream 1 — Refine existing themes

**1a. Alert-callout colors become theme-aware.** Currently `.markdown-alert-NOTE` etc. use hardcoded GitHub-light hex (`#0969da`, `#1a7f37`, `#8250df`, `#9a6700`, `#cf222e`) — they look wrong on Dracula/Nord/Solarized-Dark. Fix: introduce 10 CSS variables (`--alert-note`, `--alert-note-bg`, `--alert-tip`, …, `--alert-caution-bg`) on `:root`, override them under each dark-theme block in `themes.css`. Light themes keep the GitHub palette; dark themes get brighter, slightly desaturated versions tuned per theme.

**1b. Contrast pass on muted text.** `--fg-muted: #86868b` (light) and `#636366` (dark) fail WCAG AA at small sizes in some themes. Bump the worst offenders: Nord `--fg-muted` from `#81a1c1` (too bright/saturated for "muted") to `#7b8794`; Dracula `--fg-muted` `#6272a4` stays (it's already good). Light stays (it's fine). I'll verify each theme's `--fg`/`--fg-secondary`/`--fg-muted` against their `--bg` for AA contrast and only adjust where it actually fails.

**1c. Per-theme shadow tuning.** The light theme's layered shadows (`rgba(29,29,31,0.05)`) are invisible on dark themes. Each dark theme block already overrides `--shadow-*` — I'll verify they read well and tighten the ones that are too subtle (Dracula/Nord borders are easy to lose).

### Workstream 2 — Add 4 more themes

Add **GitHub Light**, **GitHub Dark**, **Tokyo Night**, and **Catppuccin Mocha**. (Gruvbox dropped to keep scope reasonable — 4 is already a lot.) Each gets:
- A `:root[data-theme="<id>"]` token block in `themes.css` (surfaces, text, borders, accent, code, shadows, + the 10 alert vars from 1a).
- An entry in `HLJS_FOR_THEME` in `main.js` (github → hljs-light reuse, github-dark → hljs-dark reuse, tokyo-night/catppuccin pick the closest hljs theme — tokyo-night uses `hljs-dark`, catppuccin uses `hljs-dracula` as the nearest match since hljs has no native catppuccin).
- A `<link>` hljs stylesheet in `index.html` only if a NEW hljs theme is needed; the four new app themes reuse existing hljs stylesheets, so **zero new `<link>` tags**.
- A `.theme-swatch[data-sw="<id>"]` color entry in `base.css`.
- A `<button class="ctx-item theme-item" data-theme="<id>">` in the `#theme-menu` dropdown in `index.html`.
- An `<option value="<id>">` in `#settings-theme` select in `index.html`.
- The 10 alert variables per theme (inherited from `:root` for light; overridden in each dark block).

### Workstream 3 — Reading-comfort controls

Two new settings in the settings dialog:
- **Font size** — a `<select>` (Small / Medium / Large / Extra Large) mapping to base font sizes 13 / 15 / 17 / 19px. Persisted as `mdpeek-base-font`. Default `15` (Medium, current behavior).
- **Line spacing** — a `<select>` (Compact / Normal / Relaxed) mapping to 1.5 / 1.7 / 2.0. Persisted as `mdpeek-line-height`. Default `1.7` (Normal, current behavior).

**Critical coordination with zoom:** `applyZoom()` in main.js currently uses a hardcoded `BASE_FONT_PX = 15`. I'll change it to read from the persisted base-font setting: `const BASE_FONT_PX = parseInt(localStorage.getItem('mdpeek-base-font'), 10) || 15;` — read fresh inside `applyZoom()` (not at module load, so changing the setting takes effect without a reload). Zoom then multiplies the user's chosen base by the zoom level, preserving the zoom slider's semantics. Line-height has no zoom interaction — set via a CSS variable `--content-line-height` on `:root` from JS; `.markdown-body { line-height: var(--content-line-height, 1.7) }`.
