## Plan: PDF viewing support

**Goal:** Open `.pdf` files (dialog, drag-drop, double-click) and render them cleanly inside mdpeek with pdf.js. Lazy-loaded so markdown-only users pay zero cost. PDFs are read-only (no editing).

### Architecture decisions (from research)

1. **pdf.js API + custom canvas rendering** — not the viewer iframe. Smallest, cleanest, most controllable. Uses `getDocument()` + `page.render()` into `<canvas>` elements, one per page.
2. **`convertFileSrc()` + Tauri asset protocol** to load PDF bytes — pdf.js fetches the asset URL directly via `getDocument({ url })`. This sidesteps the whole "content is a String" problem: **PDF bytes never touch `doc.content` or the Rust text-reading commands.** `doc.content` stays `''` for PDFs; the PDF is rendered from `doc.path` on demand.
3. **Lazy dynamic import** — `import('pdfjs-dist')` lives in a new `src/views/pdf-viewer.js` module that Vite code-splits into a separate chunk (~125 KB gzip main + ~300 KB worker, loaded only when a PDF tab is actually viewed). The entry bundle, startup time, and markdown path are completely unaffected.
4. **PDFs are read-only** — no edit mode, no dirty flag, no save. The mode toggle and Ctrl+E are no-ops on PDF tabs.

### Weight impact

| Component | Size | When loaded |
|---|---|---|
| pdf.js main chunk | ~125 KB gzip | Only when a PDF tab is first viewed |
| pdf.js worker | ~300 KB | Same — lazy |
| Installer | +~0.5 MB (the two chunks are bundled into `dist/`) | Always shipped, but only fetched by the webview on demand |

Installer goes from **4.01 MB → ~4.5 MB**. Still dramatically lighter than Typora/Obsidian/VS Code. The initial app load is unchanged because the chunks are lazy.

### Files touched (8)

**1. `package.json`** — add `pdfjs-dist` dependency.

**2. `src-tauri/tauri.conf.json`** — two changes:
- Enable the asset protocol + scope to allow reading any local PDF:
  ```json
  "security": {
    "csp": null,
    "assetProtocol": {
      "enable": true,
      "scope": { "allow": ["**"] }
    }
  }
  ```
  (CSP stays `null` — already permissive; no CSP directives to fight.)
- Add `pdf` to `fileAssociations`:
  ```json
  { "ext": ["pdf"], "name": "PDF", "description": "PDF document", "role": "Viewer", "mimeType": "application/pdf" }
  ```

**3. `src-tauri/capabilities/default.json`** — add `"core:default"` already present; add the asset protocol permission: `"fs:allow-asset-protocol"` (or the correct core asset permission — verified at build time).

**4. `src-tauri/src/lib.rs`** — the cold-start (`get_initial_file`) and hot-start (single-instance handler) paths both use `std::fs::read_to_string`, which fails on PDF bytes. Fix: branch on extension. For `.pdf` paths, emit `{ path, content: "" }` (empty content — the JS side detects PDF by path and uses `convertFileSrc` instead of content). No new Rust command needed — we just stop trying to read PDF bytes as text.

**5. `src-tauri/nsis/file-assoc.nsh`** — add a `mdpeek.pdf` ProgID + register `.pdf` in the install/uninstall hooks so double-clicking a PDF in Explorer offers mdpeek.

**6. `src/lib/documents.js`** — add `isPdfPath(path)` helper (`/\.pdf$/i`), and derive a `pdf` flag in `createDocument` and `restore` (same pattern as the existing `plain` flag). PDF docs get `mode: 'view'` forced, `content: ''`.

**7. `src/views/pdf-viewer.js` (NEW)** — the PDF renderer module:
- `export async function showPdf(container, filePath)` — lazy-imports pdf.js, sets the worker URL via `?url`, calls `convertFileSrc(filePath)` to get the asset URL, calls `pdfjsLib.getDocument({ url })`, renders each page into a `<canvas>` inside `container`.
- Loading + error states (a centered "Loading PDF…" spinner; a red error message if pdf.js fails to parse).
- Renders pages sequentially as you scroll (renders the first page immediately, then the rest). Each page is a `<canvas>` sized to the page at the current zoom. Re-renders on zoom changes.
- Clean teardown: `destroyPdf(container)` cancels any pending render task and clears the container.

**8. `src/main.js`** — the integration glue:
- `openPath(path, content)` — detect `isPdfPath(path)`; for PDFs, call `store.open({ path, content: '', mode: 'view' })` and skip `rewatch` (PDFs aren't watched for changes — they're read-only binary files).
- `renderActive()` — add a `doc.pdf` branch before the edit/view split: hide edit mode + mode toggle + TOC, show view mode, call `showPdf(el.document, doc.path)`. Capture `doc.scrollY` on tab switch (same as view-mode markdown).
- Drag-drop regex — add `pdf` to the accepted extensions. For dropped PDFs, skip `file.text()` (would corrupt); pass the path straight to `openPath`.
- File dialog — add a PDF filter in the Rust `open_file` command (`commands.rs`).
- The `open-file` / `get_initial_file` payloads now carry empty content for PDFs — the JS side detects PDF by path and uses `convertFileSrc` instead of content.
- Session restore — skip the `read_file` re-read for PDF paths (keep `content: ''`); PDFs restore from path alone.
- `file-changed` handler — skip PDF docs (no re-render on external change; they're binary).
- `toggleMode()` — guard `doc.pdf` (no-op, like `doc.plain`).

### The PDF viewing UX
- A `.pdf` opens as a new tab (tab title = filename, with a small "PDF" badge instead of the markdown icon).
- Pages render stacked vertically in the document pane, scrollable. The sidebar (TOC) is hidden for PDFs (no headings to list).
- Zoom (Ctrl+= / Ctrl+-) scales the rendered pages.
- Read-only — no edit mode, no save, no dirty indicator.
- Find (Ctrl+F) does not search inside PDFs (pdf.js has a text layer, but wiring it into our find module is out of scope for v1). The find bar is a no-op on PDF tabs.

### Edge cases handled
- **Large PDFs** — pages render lazily as you scroll (IntersectionObserver), not all at once.
- **Corrupt/encrypted PDF** — pdf.js throws; we show a friendly error in the container instead of crashing.
- **Cold-start with a PDF** — Rust emits empty content; JS detects by path and renders via `convertFileSrc`.
- **Session restore with a PDF tab open** — the tab restores from path alone (no content re-read).
- **Tab switching away from a PDF** — scroll position captured in `doc.scrollY`; the pdf.js document is destroyed to free memory (re-loaded on return).
- **PDF deleted on disk while open** — `convertFileSrc` fails gracefully; we show an error.

### What stays unchanged
- The markdown rendering pipeline (renderer.js, viewer.js, enhanceDom).
- The editor (editor.js, editor-logic.js).
- All existing tests (no logic changes to the pure modules).
- The find bar, themes, settings, window controls, tray.
- Installer build pipeline.

- `npm test` — 89/89 must pass (no logic changes; new module is DOM-only, not unit-tested in v1).

- Manual smoke test: open a PDF via dialog, via drag-drop, via double-click in Explorer; scroll through pages; zoom in/out; switch tabs and back (scroll position preserved); close the tab; open a markdown doc and confirm it still renders normally.

### Out of scope
- PDF find (Ctrl+F inside PDF text).
- PDF export (markdown → PDF).
- PDF annotations / form filling.
- PDF page thumbnails / outline sidebar.
