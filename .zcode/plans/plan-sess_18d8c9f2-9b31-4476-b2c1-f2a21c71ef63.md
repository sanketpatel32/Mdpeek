## Plan: Refined Windows-style window controls (top-right)

Replace the native Windows min/max/close buttons with custom HTML/CSS controls that match mdpeek's macOS-premium aesthetic, while preserving the existing close-to-tray flow.

### Why this approach
Native title-bar buttons are drawn by Windows — no CSS can touch them. The only way to restyle them is `decorations: false` + custom HTML buttons wired to Tauri's window APIs. The user picked **Refined Windows (top-right)**: minimalist `−` / `□` / `✕` glyphs in the top-right, close turns red on hover (VS Code / Edge style).

### Files touched (5)

**1. `src-tauri/tauri.conf.json`** — add `"decorations": false` to the single window object (turns off the OS title bar/controls). Also add `"shadow": true` so the window still gets a drop shadow without decorations (otherwise the window edges look flat against the desktop).

**2. `src-tauri/capabilities/default.json`** — add window permissions so JS can call the APIs:
```json
"permissions": [
  "core:default",
  "core:window:allow-minimize",
  "core:window:allow-toggle-maximize",
  "core:window:allow-unmaximize",
```

Note: close does NOT need a permission — the custom Close button calls the existing `showCloseDialog()` directly (no `.close()` call), preserving the tray/quit dialog flow unchanged.

**5. `src/main.js`** — wiring. Add at top with other imports:
```js
import { getCurrentWindow } from '@tauri-apps/api/window';
```
Add a new section (near the close-dialog code so related logic lives together):
```js
// ---------- custom window controls ----------
const appWindow = getCurrentWindow();
document.getElementById('win-minimize').addEventListener('click', () => {
  appWindow.minimize().catch((e) => console.error('minimize failed', e));
});
document.getElementById('win-maximize').addEventListener('click', async () => {
  await appWindow.toggleMaximize();
});
document.getElementById('win-close').addEventListener('click', () => {
  showCloseDialog();   // reuses existing tray/quit flow verbatim
});
// Sync the maximize/restore icon on resize (covers user maximizing via Win+Up,
// snap layouts, etc., not just our button).
const icoMax = document.querySelector('.win-ico-max');
const icoRestore = document.querySelector('.win-ico-restore');
async function syncMaxIcon() {
  const maximized = await appWindow.isMaximized();
  icoMax.classList.toggle('hidden', maximized);
  icoRestore.classList.toggle('hidden', !maximized);
}
appWindow.onResized(syncMaxIcon);
syncMaxIcon();  // initial state
```

### What stays unchanged
- The Rust `CloseRequested` interception in `lib.rs` (lines 109–117) — still emits `close-requested` and prevents close. Needed because taskbar right-click → Close, Alt+F4, and tray quit still go through the OS event path.
- `hide_to_tray` / `quit_app` Rust commands.
- The entire close-dialog modal, the `close-requested` listener, `showCloseDialog`, `resolveClose`, the "remember preference" checkbox.
- The version pill, theme dropdown, tabs, toolbar — all unaffected.
- `core:default` already in permissions.

### Decisions worth flagging
- **Minimize = taskbar minimize**, NOT hide-to-tray. The custom Minimize button does `appWindow.minimize()` (window goes to the taskbar). This matches what a Windows user expects the `−` button to do. Hide-to-tray stays accessible via the **Close** button → dialog → "Minimize to tray", unchanged. (This is a deliberate semantic split: − means taskbar, ✕ means tray-or-quit.)
- **Maximize uses `toggleMaximize()`** — single button, icon swaps between box and restore glyphs via the `tauri://resize` listener so it stays correct even when the user maximizes via Win+Up or snap.
- **Drag region on the whole header** — `data-tauri-drag-region` on `<header>`. Tauri automatically excludes buttons/inputs/links from drag, so tabs and toolbar keep working. The window controls get an explicit `-webkit-app-region: no-drag` as a safety net.
