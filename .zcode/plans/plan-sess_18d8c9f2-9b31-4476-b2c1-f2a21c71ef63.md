## Plan: Markdown engine overhaul (4 areas)

All four improvements land in `src/lib/renderer.js`, `src/styles/content.css`, plus new tests. One new dependency (`marked-footnote`, ~3KB) for correct footnote parsing — writing a footnote tokenizer by hand is error-prone and this package is tiny and well-maintained.

### 1. Footnotes + heading IDs (`renderer.js`)

**Heading IDs** — add a custom `heading` renderer that slugifies the text into a GitHub-compatible id (lowercase, spaces→hyphens, strip non-alphanumeric, dedupe via a counter). No dependency needed; marked v18 lets you override `renderer.heading`. This also makes in-document `#section` anchor links work natively, and `buildToc()` in viewer.js can stop assigning its own `h-N` ids (it'll use the slugified ones, falling back to assignment only when empty).

**Footnotes** — add `marked-footnote` package. Fixes the currently-broken `text[^1]` syntax (today it produces a garbage link to literal "note"). Adds proper `[^1]` reference rendering + a definitions section at the bottom. CSS in section 2.

### 2. Task lists + alert callouts + footnote CSS (`content.css`)

- **Task lists** — style `li > input[type=checkbox]`: custom checkbox appearance (accent-colored border, check mark when checked), and `li:-webkit-handler-checked` gets muted/strikethrough text (GitHub-style "done" look).
- **Alert callouts** — marked v18 has GFM alerts built-in (enabled via the `alerts` extension option, off by default). Enable it. Renders `> [!NOTE]` / `[!TIP]` / `[!IMPORTANT]` / `[!WARNING]` / `[!CAUTION]` as `<blockquote class="markdown-alert markdown-alert-NOTE">`. CSS: colored left-border + tinted background per type (blue/note, green/tip, purple/important, orange/warning, red/caution), plus a leading icon + title line.
- **Footnotes** — style the `sup` reference (small, accent-colored, bracketed) and the footnotes section (top border, smaller font, muted).

### 3. More syntax-highlight languages (`renderer.js`)

Keep the `highlight.js/lib/common` entry chunk (36 langs, already loaded). **Dynamically register** a handful of high-value extras on first use via `highlight.js/lib/languages/<lang>` dynamic import — candidates: `dockerfile`, `ini`, `properties`, `toml`, `makefile`, `latex`, `nginx`, `diff`. When a fenced code block names one of these and it's not yet registered, dynamically import + `hljs.registerLanguage()`, then highlight. First use of that language has a ~one-frame delay; subsequent uses are instant. A `Set` tracks what's registered so we never re-import. Falls back to plaintext if the import fails. This adds zero KB to the entry chunk.

### 4. Security + perf hardening (`renderer.js`)

- **DOMPurify link hardening** — add a `afterSanitizeAttributes` hook that sets `target="_blank"` + `rel="noopener noreferrer"` on every `<a>` with an `href` (external links open safely in the system browser via the existing opener-plugin click handler; the attributes are belt-and-suspenders for defense in depth). One-time hook registration at module load.
- **Render cache** — a `Map<string, string>` LRU capped at 64 entries (oldest evicted on overflow). Keyed by the raw markdown string. `renderMarkdown()` checks the cache before parsing. Big win for tab-switch re-renders (same content re-rendered) and for the edit-mode debounced preview (identical intermediate states). Invalidateation is automatic — the cache key is the content itself, so editing produces a new key.
- **Lazy mermaid** — already lazy (dynamic import). No change.

### Files touched (4) + 1 new dependency

1. **`src/lib/renderer.js`** — the bulk of the work:
   - Import + register `marked-footnote`
   - Add `heading` renderer with slugify + dedupe counter (reset per `renderMarkdown` call via a module-level `Map`)
   - Enable GFM alerts in the `marked.use()` options
   - Add dynamic language registration for the extra hljs languages
   - Add DOMPurify `afterSanitizeAttributes` hook (registered once, guarded by a flag)
   - Add the render cache (`Map`, cap 64, LRU eviction)

2. **`src/styles/content.css`** — new CSS for:
   - Task list checkboxes (custom box + checked strikethrough)
   - Alert callouts (5 variants with colored borders/tints/icons)
   - Footnote references + definitions section

3. **`src/views/viewer.js`** — `buildToc()` updated to use the heading's existing slug id when present (instead of always assigning `h-N`). Keeps the fallback for headings without text.

4. **`test/renderer.test.js`** — new tests:
   - Heading IDs: `renderMarkdown('## Hello World')` contains `id="hello-world"`
   - Heading ID dedup: two identical headings get distinct ids (`hello`, `hello-2`)
   - Footnotes: `text[^1]` + definition renders a `<a>` ref AND a definitions section (not the current garbage link)
   - Task list: `- [x] done` contains a checked checkbox
   - Alert callout: `> [!NOTE]` renders `<blockquote class="markdown-alert">`
   - External links get `target="_blank"` + `rel="noopener noreferrer"`
   - Cache: calling `renderMarkdown` twice with same input returns identical output (and is fast)
   - All existing tests keep passing unchanged

5. **`package.json`** — add `marked-footnote` to dependencies (~3KB gzipped).

### What stays unchanged
- KaTeX math rendering (already working via marked-katex-extension)
- Mermaid diagram routing (the `mermaid` lang check stays first in the code renderer)
- Copy buttons on code blocks
- XSS stripping of `javascript:` / `onerror` / `<script>` (DOMPurify defaults — the new hook only *adds* attributes, never weakens)
- The 81 existing tests (the renderer tests are additive; documents/editor-logic tests untouched)

### Edge cases
- **Heading slug collisions**: two `## Introduction` headings → `introduction`, `introduction-2` (GitHub behavior). Counter resets per render call.
- **Empty heading text**: `## ` → falls back to `h-N` assignment in buildToc (no slug).
- **Dynamic language import fails** (offline): falls back to plaintext highlighting, no error thrown.
- **Cache + DOMPurify**: the cache stores the *post-sanitize* HTML, so cached output is safe by construction.
- **Alert callout inside list**: marked's alerts extension handles blockquote nesting correctly; CSS scopes to `.markdown-alert` so it won't leak.

### Verification
- `npm test` — 81 existing + ~7 new tests = ~88 tests, all passing.
- `npm run tauri:build` — confirms the new dep bundles cleanly.
- Manual smoke test: render a doc with footnotes, a `> [!WARNING]` callout, a `- [x]` task list, a Dockerfile code block, and a `## Section` heading — click a TOC entry and confirm it scrolls; click an in-doc `[#section]` link.
- Release as **v0.5.0** (MINOR — significant feature additions).

### Out of scope
- Custom markdown syntax extensions (e.g., emojis, definition lists, spoilers)
- Find-and-replace
- Markdown linting / formatting
- Switching to a different parser (remark, markdown-it) — marked v18 is fine.
